# 1.3.0 — Decision record

Durable record of the decisions and rationale behind the 1.3.0 release, so the "why" survives now
that the release's scratch planning files are gone. Each section is *what was decided* and *why*.
For the next-major backlog see [`2.0.0.md`](2.0.0.md).

## Scope

1.3.0 mirrors the new Rust 1.3.0 chart-trend functions and pays down DX/docs/packaging debt. The
wrapper layer had **no calculation or wiring bugs** (confirmed by a read-only audit across all nine
binding modules), so the release is a **semver-clean minor**: additive bindings + documentation +
metadata + typing only. What shipped:

- `chart_trends.peak_favorable_move` / `valley_favorable_move` bindings.
- Python regression tests for the inherited Rust 1.3.0 degenerate-input fixes.
- README accuracy pass + docstring keyword-`TypeError` fixes.
- String-alias documentation + tests (the three accepted vocabularies, pinned).
- Package-metadata cleanup (Documentation URL, classifiers 3.10–3.14).
- Type stubs (`.pyi`) + `py.typed` + `__version__` via a mixed package layout.
- `cargo fmt --check` added to CI; `docs/2.0.0.md` breaking-change backlog.

**Out of scope (decided, not deferred): binding the Rust `basic_indicators` module.** It is a
statistical-primitives surface (mean / median / mode / variance / std / abs-dev / min / max /
quantile). Python's `statistics` and `math` provide these natively; re-binding through PyO3 would
hand users a slower, FFI-marshalled version of the standard library. Documented in the README to
pre-empt the recurring "module parity" question.

## Release gate & favorable-move semantics

The Rust crate `centaur_technical_indicators 1.3.0` was already published before this release began,
so the binding bump was step one (no gating phase). Favorable-move semantics, verified against the
1.3.0 source:

- Forward window is the **inclusive** range `[index + 1, index + period]`; results are **signed**
  (not floored). `peak_favorable_move = prices[index] − min(window)`;
  `valley_favorable_move = max(window) − prices[index]`.
- In-bounds **iff** `index + period < len(prices)`. Errors map to `ValueError`: empty → `EmptyData`;
  `period == 0` and out-of-bounds window → `InvalidPeriod`.

## Inherited-fix inventory

The dependency bump to 1.3.0 was **not** dependency-only — it inherited user-visible behavior fixes,
now observable through the bindings and documented in the changelog:

- `chart_trends.peaks` / `valleys`: corrected for an extremum at index 0 and for one retained after a
  monotonic run.
- `trend_indicators.single.aroon_up` / `aroon_down` and
  `momentum_indicators.single.stochastic_oscillator`: return `NaN` instead of panicking on all-NaN
  input.
- Cauchy IQR-scale hardening is reachable only via `deviation_model="cauchy"` (there is **no**
  `cauchy_iqr_scale` binding); it is exercised indirectly through
  `candle_indicators.single.moving_constant_bands` on degenerate input.

The regression tests for these live at the FFI boundary (a Rust panic crossing FFI is observable in
Python); exhaustive extremum testing remains the Rust repo's job. All exact-value expectations were
generated by running against the installed 1.3.0 crate, never hand-computed.

## DX & packaging

- **`__version__` uses the mixed layout + `importlib.metadata`.** The original plan to
  `m.add("__version__", env!("CARGO_PKG_VERSION"))` on the inner module does **not** survive maturin's
  auto-generated `from .ext import *` (dunders are not re-exported), so `cti.__version__` would have
  been absent. The fix adopts a mixed layout (`[tool.maturin] python-source = "python"`) with a
  hand-authored `__init__.py` that replicates maturin's shim, preserves `__doc__`/`__all__`, and
  resolves `__version__` from the installed dist-info (derived from `Cargo.toml [package] version`).
- **Submodule import paths.** `from .ext import *` only binds submodules as *attributes*, so the
  `.pyi` stubs (shipped as package submodules) advertised `import …<sub>` paths that failed at
  runtime. `__init__.py` registers each submodule in `sys.modules` under its qualified name so the
  advertised paths resolve; `tests/test_package_layout.py` guards the contract.
- **Stale `python/` cleanup.** The pre-existing `python/centaur_technical_indicators/` held a stale,
  gitignored `.so`; it was repurposed as the mixed-layout source root (the rebuilt extension stays
  gitignored).
- **`.pyi` stubs were generated, not hand-written:** parameter names from each function's
  `__text_signature__`, return types from the Rust `PyResult<…>`, then **cross-checked against the
  runtime surface** (exact match — 97 functions across the nine submodules, 0 missing/extra).
- **`__all__` is PyO3-auto-populated** with the nine submodules — verification-only, no code added.
- The internal `docs/` directory is excluded from the sdist (`[tool.maturin] exclude`); it never
  enters the wheel.

## Quality

- **S0 `cargo fmt` baseline.** The committed Rust source failed `cargo fmt --all -- --check` on
  pre-existing drift; because that gate is whole-crate, no later session could pass its own formatting
  gate until the drift was cleared. A single repo-wide reformat established the baseline (the one
  acknowledged exception to the no-opportunistic-refactor rule), and `cargo fmt --check` became a CI
  gate.
- **Indicator-count recount:** ~55 distinct indicator names → the README/metadata claim moved from
  "60+" to a defensible "50+".
- **Docstring keyword `TypeError`s:** PyO3 exposes parameters as keyword arguments under the Rust
  parameter names, so a docstring naming the wrong keyword raises a real `TypeError`. Fixed:
  `supertrend` (single + bulk) `constant_type_model` → `constant_model_type`; `true_range` (single)
  `previous_close` → `close`.
- **Review catches (Codex):**
  - #40 — the six `chart_trends` `See:` links pointed at docs pages that 404 (the chart-trends docs
    are not yet published; existing module pages return 200). They were **deferred**, to be added when
    the pages exist. Also #40 — `ulcer_index` has **both** `single` and `bulk` variants (only
    `volatility_system` is bulk-only); an earlier planning note wrongly called it single-only.
  - #41 — the `.pyi` stubs advertised submodule import paths that raised `ModuleNotFoundError` at
    runtime (the `sys.modules` fix above).

## 2.0 deferrals

Breaking changes are not actionable in 1.x; they are logged in [`2.0.0.md`](2.0.0.md): the misleading
`sma`/`ma` model-type aliases, the TSI single/bulk argument-order mismatch, deprecated upstream
functions exposed without a Python deprecation signal, and the Python-unreachable parameterized enum
variants. 1.3.0 documents and pins the current behavior; 2.0 changes it.

## Open sub-decisions (resolved)

- **New disambiguating model-type aliases?** No — 1.3.0 documents and pins the existing set only
  (the `sma`→Smoothed / `ma`→Simple inversion is real and now explicit in the docs/tests).
- **`cargo clippy -D warnings` in CI?** Parked — risk of macro-generated lint noise on a PyO3 crate;
  not added without maintainer approval.
